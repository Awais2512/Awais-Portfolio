# LLM Inference Optimization: Lessons from Building FinanceGPT

When we started building FinanceGPT, our early prototype was burning through API credits at $800/month with 4+ second response times. By launch, we had cut costs to $320/month and latency to under 2 seconds. Here's exactly what we did.

## The Cost Math First

Before optimizing, measure. Here's the formula I use to estimate LLM API costs:

```
Monthly cost = (avg_input_tokens × input_price + avg_output_tokens × output_price)
               × requests_per_day × 30
```

For GPT-4 Turbo at our traffic:
- Avg input: 2,400 tokens × $0.01/1K = $0.024
- Avg output: 800 tokens × $0.03/1K = $0.024
- Total per request: $0.048
- At 500 requests/day × 30 days = **$720/month**

Once you have this, you can target the biggest levers.

## Lever 1: Prompt Caching

OpenAI's prompt caching gives 50% discount on cached input tokens for prompts over 1,024 tokens. The key is structuring your prompts so the static prefix is long and stable:

```python
SYSTEM_PROMPT = """
You are FinanceGPT, an expert financial analysis assistant...
[800+ tokens of detailed instructions and examples]
"""

def build_prompt(user_query: str, context: str) -> list:
    return [
        {"role": "system",    "content": SYSTEM_PROMPT},   # Cached after first call
        {"role": "user",      "content": f"Context:\n{context}\n\nQuery: {user_query}"},
    ]
```

The system prompt gets cached automatically. This saved us ~30% on input token costs immediately.

## Lever 2: Model Routing

Not every query needs GPT-4. We categorized queries into tiers:

| Tier | Query Type | Model | Cost |
|------|-----------|-------|------|
| 1 | Simple lookups, definitions | GPT-3.5-turbo | $0.002 |
| 2 | Data analysis, comparisons | GPT-4-turbo-mini | $0.010 |
| 3 | Complex reasoning, synthesis | GPT-4 | $0.048 |

A lightweight classifier routes queries before they hit the main model:

```python
async def route_query(query: str) -> str:
    # Fast tier-1 check: keyword matching
    simple_patterns = ["what is", "define", "list the", "when was"]
    if any(query.lower().startswith(p) for p in simple_patterns):
        return "gpt-3.5-turbo"

    # Tier-2 check: short LLM call
    tier = await classify_complexity(query)  # GPT-3.5 call, ~100 tokens
    return "gpt-4-turbo" if tier == "complex" else "gpt-4-turbo-mini"
```

Result: 60% of queries routed to cheaper models, 25% cost reduction.

## Lever 3: Streaming for Perceived Latency

Users perceive streaming responses as faster even when total latency is the same. First token typically arrives in 300–500ms:

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse

@app.post("/analyze")
async def analyze(request: AnalysisRequest):
    async def generate():
        async for chunk in client.chat.completions.create(
            model="gpt-4-turbo",
            messages=build_prompt(request.query, request.context),
            stream=True,
        ):
            delta = chunk.choices[0].delta.content
            if delta:
                yield f"data: {delta}\n\n"
        yield "data: [DONE]\n\n"

    return StreamingResponse(generate(), media_type="text/event-stream")
```

This eliminated "waiting" perception entirely — user satisfaction scores improved noticeably.

## Lever 4: Response Caching

For deterministic queries (same input → same output), cache aggressively:

```python
import hashlib
import redis

cache = redis.Redis(host="localhost", port=6379, db=0)
CACHE_TTL = 3600  # 1 hour

def cache_key(query: str, context_hash: str) -> str:
    return hashlib.sha256(f"{query}:{context_hash}".encode()).hexdigest()

async def get_analysis(query: str, context: str) -> str:
    key = cache_key(query, hashlib.md5(context.encode()).hexdigest())

    cached = cache.get(key)
    if cached:
        return cached.decode()

    response = await call_llm(query, context)
    cache.setex(key, CACHE_TTL, response)
    return response
```

Financial data queries for the same report repeated frequently. Cache hit rate reached 35%, cutting those calls to zero.

## Lever 5: Batching for Non-Real-Time Workloads

For batch jobs (nightly reports, bulk analysis), use async batching to maximize throughput:

```python
import asyncio

async def process_batch(queries: list[str]) -> list[str]:
    semaphore = asyncio.Semaphore(10)  # Max 10 concurrent requests

    async def process_one(query):
        async with semaphore:
            return await call_llm(query)

    return await asyncio.gather(*[process_one(q) for q in queries])
```

## What We Didn't Do (Yet)

- **Self-hosted models**: Hosting Llama 3 would cut API costs further but adds infrastructure complexity. Worth it at 10× our current traffic.
- **Quantized models**: For preprocessing tasks, 4-bit quantized models on GPU are cost-effective.
- **Semantic caching**: Using embedding similarity for cache lookup instead of exact hash matching — catches paraphrased queries.

Optimization is a continuous process. Measure, find the biggest lever, pull it, measure again.
